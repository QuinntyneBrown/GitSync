# L2 - Detailed Requirements

## L2-001: .NET Global Tool Packaging
**Traces to:** L1-001

The `.csproj` file shall set `<PackAsTool>true</PackAsTool>` and declare a `<ToolCommandName>` (e.g., `gitsync`) so the tool is installed as a shell-invocable command. The `OutputType` shall be `Exe`.

**Acceptance Criteria:**
1. Given the project is packed with `dotnet pack`, when the resulting `.nupkg` is installed with `dotnet tool install -g`, then the command name defined in `<ToolCommandName>` is available on the system PATH.
2. Given the tool is installed globally, when the user runs the tool command name with no arguments, then the default sync command executes without error.
3. Given the project is built with `dotnet build`, when the output type is inspected, then it is an executable (`Exe`).

---

## L2-002: Command-Per-File Pattern
**Traces to:** L1-001

Each `System.CommandLine` `Command` subclass shall be defined in its own dedicated `.cs` file. No single file shall define more than one `Command` subclass. A root command file shall wire up all subcommands.

**Acceptance Criteria:**
1. Given the source tree is inspected, when command class files are listed, then each file contains exactly one class that extends `Command` (or `RootCommand`).
2. Given a new command needs to be added, when it is placed in its own file and registered in the root command, then the existing commands are unaffected and all commands function correctly.
3. Given the project compiles, when the root command is constructed, then all child commands are registered without duplication.

---

## L2-003: System.CommandLine Integration
**Traces to:** L1-001

The tool shall use the `System.CommandLine` NuGet package for all CLI argument and option parsing. Options and arguments shall be declared as typed `Option<T>` instances attached to their respective `Command`. Command handlers shall be wired via `SetHandler`.

**Acceptance Criteria:**
1. Given the tool is invoked with `--help`, when the help text is rendered, then all declared options appear with their descriptions and default values.
2. Given an option is passed with an invalid type (e.g., a string where a `TimeSpan` is expected), when the tool parses arguments, then it exits with a non-zero exit code and prints a descriptive parse error to stderr.
3. Given no options are provided, when the default command runs, then all options resolve to their documented default values.

---

## L2-004: Default Sync Command
**Traces to:** L1-002

The sync loop command shall be the default command of the tool — i.e., it executes when the tool is invoked with no subcommand name. It shall be defined as the `RootCommand` handler or as the only subcommand set as the default.

**Acceptance Criteria:**
1. Given the tool is invoked with no subcommand name but with valid options, when it starts, then the sync loop begins executing.
2. Given the tool is invoked with `--help`, when help is displayed, then the default command's purpose, all its options, and their defaults are clearly described.
3. Given the tool is invoked with an unrecognized subcommand name, when it parses arguments, then it exits with a non-zero exit code and prints a parse error.

---

## L2-005: Configurable Poll Interval
**Traces to:** L1-002

The sync command shall expose an `--interval` option of type `TimeSpan`. The default value shall be `00:00:01` (1 second). The minimum accepted value shall be `00:00:00.100` (100 milliseconds). Values below the minimum shall be rejected at parse time.

**Acceptance Criteria:**
1. Given no `--interval` is specified, when the loop runs, then the delay between iterations is 1 second.
2. Given `--interval 00:00:05` is specified, when the loop runs, then the delay between iterations is 5 seconds.
3. Given `--interval 00:00:00.050` (50 ms, below minimum) is specified, when the tool parses arguments, then it exits with a non-zero exit code and emits an error message referencing the minimum allowed value.
4. Given `--interval 00:00:00.100` (100 ms, at minimum boundary) is specified, when the tool parses arguments, then it is accepted and the loop runs at 100 ms intervals.

---

## L2-006: Configurable End Time
**Traces to:** L1-002

The sync command shall expose an `--end-time` option of type `DateTimeOffset` (ISO 8601 format). When provided, the loop shall stop after the current iteration completes if the current UTC time is at or past the specified end time.

**Acceptance Criteria:**
1. Given `--end-time` is set to a time 5 seconds in the future, when the loop runs, then iterations stop once UTC now >= the specified end time, and the tool exits with exit code `0`.
2. Given `--end-time` is set to a time in the past, when the tool starts, then no iterations execute, a warning is logged, and the tool exits with exit code `0`.
3. Given `--end-time` is provided with an invalid date string, when the tool parses arguments, then it exits with a non-zero exit code and prints a descriptive parse error.
4. Given both `--end-time` and `--duration` are specified, when the tool parses arguments, then it exits with a non-zero exit code indicating the options are mutually exclusive.

---

## L2-007: Configurable Duration with Default
**Traces to:** L1-002

The sync command shall expose a `--duration` option of type `TimeSpan`. When provided, the loop shall run for at most that duration from the moment execution begins. When neither `--end-time` nor `--duration` is provided, the tool shall default to a duration of 8 hours (`08:00:00`).

**Acceptance Criteria:**
1. Given neither `--end-time` nor `--duration` is specified, when the loop starts, then it is scheduled to stop after 8 hours from start time.
2. Given `--duration 00:01:00` is specified, when the loop starts, then it stops after 1 minute from start time.
3. Given `--duration 00:00:00` (zero duration) is specified, when the tool starts, then no iterations execute, a warning is logged, and the tool exits with exit code `0`.
4. Given `--duration -00:00:01` (negative duration) is specified, when the tool parses arguments, then it exits with a non-zero exit code and prints a descriptive parse error.

---

## L2-008: Loop Iteration Timing and Graceful Shutdown
**Traces to:** L1-002

The loop shall use a cancellation-aware delay (`Task.Delay` with `CancellationToken`) between iterations to avoid CPU spinning. When Ctrl+C is pressed, the cancellation token shall be signalled, the current iteration shall be allowed to finish (or aborted if mid-delay), and the tool shall exit cleanly with exit code `0`.

**Acceptance Criteria:**
1. Given the loop is running, when Ctrl+C is pressed mid-delay, then the current delay is cancelled, no new iteration starts, and the process exits with exit code `0`.
2. Given the loop is running and Ctrl+C is pressed while a git operation is in progress, when the git process completes or its timeout fires, then the loop exits cleanly without leaving orphaned git processes.
3. Given the loop is running, when the end time or duration is reached after an iteration completes, then no new iteration starts, a completion message is logged, and the process exits with exit code `0`.

---

## L2-009: Git Executable Discovery
**Traces to:** L1-003

The tool shall locate the `git` executable by resolving it from the system `PATH` at startup. If `git` is not found on `PATH`, the tool shall log a clear error message and exit with a non-zero exit code before any loop iteration begins.

**Acceptance Criteria:**
1. Given `git` is installed and on `PATH`, when the tool starts, then it resolves the git executable path and logs it at `Debug` level.
2. Given `git` is not found on `PATH`, when the tool starts, then it logs an error message stating git was not found and exits with exit code `1` before executing any loop iteration.
3. Given the resolved git path exists at startup but is deleted before the first iteration, when the first git command runs, then the error is caught, logged, and treated as an iteration error per L2-031.

---

## L2-010: Git Pull Execution per Iteration
**Traces to:** L1-003

At the start of each iteration, the tool shall execute `git pull` in the configured working directory. The standard output and standard error of the git process shall be captured and logged. If `git pull` exits with a non-zero code, the error shall be logged and the remainder of the iteration (change detection and commit) shall be skipped.

**Acceptance Criteria:**
1. Given the working directory is a clean git repository with a remote, when `git pull` succeeds (exit code `0`), then the stdout is logged at `Debug` level and the iteration continues to change detection.
2. Given the remote is unreachable, when `git pull` fails (non-zero exit), then stderr is logged at `Warning` level and the iteration skips commit/push but does not crash the loop.
3. Given `git pull` produces merge conflict output, when it exits non-zero, then the conflict output is logged at `Warning` level and the iteration is skipped.
4. Given `git pull` is running and the git command timeout expires, when the process is killed, then a timeout warning is logged and the iteration is skipped.

---

## L2-011: Change Detection
**Traces to:** L1-004

After a successful `git pull`, the tool shall execute `git status --porcelain` in the working directory. If the output is non-empty, the repository is considered to have uncommitted local changes. If the output is empty, the iteration ends without committing.

**Acceptance Criteria:**
1. Given the working directory has untracked or modified files, when `git status --porcelain` is executed, then its output is non-empty and the tool proceeds to staging.
2. Given the working directory is clean (no changes), when `git status --porcelain` is executed, then its output is empty, a `Debug` log entry records "no changes detected", and the tool skips staging and commit.
3. Given `git status` exits with a non-zero code, when the error occurs, then it is logged at `Warning` level and the iteration is skipped.

---

## L2-012: Stage All Changes
**Traces to:** L1-004

When local changes are detected, the tool shall execute `git add -A` in the working directory to stage all tracked and untracked changes.

**Acceptance Criteria:**
1. Given changes are detected, when `git add -A` is executed, then it exits with code `0` and the tool proceeds to commit.
2. Given `git add -A` exits with a non-zero code, when the error occurs, then it is logged at `Error` level and the iteration skips commit and push.
3. Given a file path contains special characters, when `git add -A` is executed, then it stages all files without error (the `-A` flag handles path quoting).

---

## L2-013: Commit with Message
**Traces to:** L1-004

After staging, the tool shall execute `git commit -m "<message>"` using the resolved commit message. The commit message template shall have any `{timestamp}` placeholder replaced with the current UTC date/time in ISO 8601 format before use.

**Acceptance Criteria:**
1. Given the default commit message template is used, when a commit is created, then the commit message contains the UTC timestamp in ISO 8601 format.
2. Given `--message "deploy: {timestamp}"` is specified, when a commit is created, then the commit message is `deploy: <current UTC ISO 8601 timestamp>`.
3. Given `git commit` exits with a non-zero code (e.g., nothing to commit), when the error occurs, then it is logged at `Warning` level and push is skipped.
4. Given the commit message after template substitution is empty or whitespace-only, when the tool resolves the message, then it substitutes the default message and logs a warning.

---

## L2-014: Push to Origin
**Traces to:** L1-004

After a successful commit, the tool shall execute `git push` in the working directory to push to the tracked upstream. If no upstream is configured, the push shall be attempted as `git push origin HEAD`.

**Acceptance Criteria:**
1. Given a commit was successfully created, when `git push` executes, then it exits with code `0` and an `Information` log entry records the push as successful.
2. Given `git push` fails (non-zero exit), when the error occurs, then stderr is logged at `Error` level, the iteration is marked as failed, and the loop continues.
3. Given no upstream branch is configured, when `git push` fails, then the tool retries with `git push origin HEAD` and logs the retry at `Debug` level.
4. Given the push succeeds, when the iteration ends, then an `Information` log entry summarizes: pulled, N files changed, committed, pushed.

---

## L2-015: Skip Commit/Push When No Changes
**Traces to:** L1-004

When `git status --porcelain` returns empty output, the tool shall not execute `git add`, `git commit`, or `git push`. It shall log at `Debug` level that the iteration completed with no changes.

**Acceptance Criteria:**
1. Given a clean repository, when an iteration runs, then no git add, commit, or push commands are executed.
2. Given a clean repository, when an iteration runs, then the log contains a `Debug` entry: "Iteration complete — no changes to commit."
3. Given multiple consecutive clean iterations, when the loop runs, then no commit history is created on the remote for those iterations.

---

## L2-016: Working Directory Option
**Traces to:** L1-005

The sync command shall expose a `--path` option of type `DirectoryInfo`, defaulting to the current working directory of the calling process. The resolved path shall be validated before the loop starts.

**Acceptance Criteria:**
1. Given no `--path` is specified, when the tool starts, then all git operations use the current working directory.
2. Given `--path /some/repo` is specified, when the tool starts, then all git operations are performed in `/some/repo`.
3. Given `--path` points to a directory that does not exist, when the tool starts, then it logs an error and exits with exit code `1` before any loop iteration.
4. Given `--path` points to a directory that exists but is not a git repository (no `.git` entry), when the tool starts, then it logs an error and exits with exit code `1` before any loop iteration.

---

## L2-017: Commit Message Template Option
**Traces to:** L1-005

The sync command shall expose a `--message` option of type `string`. The default value shall be `"Auto-sync: {timestamp}"`. The `{timestamp}` token shall be replaced at commit time with the current UTC time in `yyyy-MM-ddTHH:mm:ssZ` format.

**Acceptance Criteria:**
1. Given no `--message` is specified, when a commit is made, then the commit message is `Auto-sync: <yyyy-MM-ddTHH:mm:ssZ>`.
2. Given `--message "chore: auto commit {timestamp}"` is specified, when a commit is made, then the commit message is `chore: auto commit <timestamp>`.
3. Given `--message` is an empty string, when the tool starts, then it logs a warning and substitutes the default message.
4. Given the `--message` value contains shell-injection characters (e.g., backticks, semicolons, `$()`), when the commit is executed, then the message is passed as a literal argument to the git process (not via shell interpolation) so injection is not possible.

---

## L2-018: Log Level Option
**Traces to:** L1-005

The sync command shall expose a `--log-level` option of type `Microsoft.Extensions.Logging.LogLevel` (string-parsed). The default shall be `Information`. Valid values are: `Trace`, `Debug`, `Information`, `Warning`, `Error`, `Critical`, `None`.

**Acceptance Criteria:**
1. Given no `--log-level` is specified, when the tool runs, then it logs at `Information` level and above.
2. Given `--log-level Debug` is specified, when the tool runs, then `Debug` and above messages are emitted.
3. Given `--log-level Warning` is specified, when a successful iteration completes, then no `Information` or `Debug` entries are emitted for that iteration.
4. Given an invalid log level string (e.g., `--log-level Verbose`), when the tool parses arguments, then it exits with a non-zero exit code and lists valid values.

---

## L2-019: Structured Console Logging
**Traces to:** L1-006

The tool shall configure `Microsoft.Extensions.Logging` with a console provider via `AddConsole()`. Log entries shall include timestamp, log level, category (class name), and message. Each git command invocation, result, and iteration summary shall be logged.

**Acceptance Criteria:**
1. Given the tool runs at `Information` level, when an iteration executes a pull, detects changes, commits, and pushes, then four `Information` or higher entries appear in the console output for that iteration.
2. Given a git command produces stderr output, when it is captured, then the stderr content is included in the log entry (not silently discarded).
3. Given the tool starts, when the first log entry appears, then it includes the tool name, configured options summary (path, interval, end time), and log level.
4. Given the log level is `None`, when the tool runs, then no console output is produced.

---

## L2-020: DI Service Registration
**Traces to:** L1-007

All application services (git executor, change detector, commit service, loop orchestrator) shall be registered in `IServiceCollection` and resolved via constructor injection. No `new` keyword shall be used to instantiate services outside of DI registration or factory lambdas.

**Acceptance Criteria:**
1. Given the DI container is built, when all required services are resolved, then no `InvalidOperationException` for missing registrations is thrown.
2. Given a service depends on `ILogger<T>`, when it is resolved from the DI container, then a properly configured logger is injected.
3. Given the tool uses `IHost`, when `host.RunAsync()` or `host.StartAsync()` is called, then all registered hosted services start correctly.

---

## L2-021: System.CommandLine and DI Integration
**Traces to:** L1-007

Command handlers shall access DI-registered services by accepting them as `SetHandler` parameters resolved from the service provider, or via a middleware/binder pattern. Commands shall not directly instantiate services.

**Acceptance Criteria:**
1. Given a command handler requires a `IGitService`, when the handler is invoked, then the service is resolved from the DI container and injected.
2. Given the DI container has a service registered with a specific lifetime (e.g., `Singleton`), when the command handler resolves it, then the lifetime semantics are respected.

---

## L2-022: NuGet Package Metadata
**Traces to:** L1-008

The `.csproj` shall declare the following NuGet metadata properties: `PackageId`, `Version`, `Description`, `Authors`, `PackageTags`, `PackageLicenseExpression` (or `PackageLicenseFile`), and `RepositoryUrl`. The `PackageId` shall be the intended NuGet package name.

**Acceptance Criteria:**
1. Given the project is packed with `dotnet pack`, when the `.nupkg` is inspected, then all required metadata fields are present and non-empty.
2. Given the `Version` follows semantic versioning (`MAJOR.MINOR.PATCH`), when the package is published, then the version appears correctly on NuGet.org.
3. Given `PackageTags` is set, when the package is published, then it is discoverable via those tags on NuGet.org.

---

## L2-023: Semantic Versioning
**Traces to:** L1-008

The tool shall use semantic versioning (`MAJOR.MINOR.PATCH`). The version shall be defined once in the `.csproj` (or a `Directory.Build.props` file) and shall not be hard-coded elsewhere. Pre-release versions shall use the `-alpha`, `-beta`, or `-rc.N` suffix convention.

**Acceptance Criteria:**
1. Given the version is `1.0.0`, when the tool is run with `--version`, then it outputs `1.0.0`.
2. Given the version is updated in the `.csproj`, when the project is packed, then the `.nupkg` filename and package metadata reflect the new version.
3. Given a pre-release version `1.0.0-beta.1` is set, when the package is published with `--prerelease`, then it is available as a pre-release on NuGet.org.

---

## L2-024: Git Command Timeout
**Traces to:** L1-009

Each git subprocess shall be subject to a configurable timeout. The timeout shall be exposed via a `--git-timeout` option of type `TimeSpan`, defaulting to `00:00:30` (30 seconds). If a git process does not exit within the timeout, it shall be killed and the iteration shall be treated as failed.

**Acceptance Criteria:**
1. Given the default timeout of 30 seconds, when a git command completes in under 30 seconds, then it is not killed.
2. Given `--git-timeout 00:00:05` is specified and a git command hangs for more than 5 seconds, when the timeout fires, then the process is forcefully terminated and a `Warning` log entry records the timeout.
3. Given a git process is killed due to timeout, when the iteration ends, then the loop continues to the next iteration after the configured interval.

---

## L2-025: Per-Iteration Error Isolation
**Traces to:** L1-009

Any unhandled exception thrown during an iteration (git execution failure, file system error, etc.) shall be caught at the loop level, logged at `Error` level with the full exception details, and must not propagate to crash the process. The loop shall continue to the next iteration.

**Acceptance Criteria:**
1. Given a git command throws an unexpected exception (e.g., `IOException`), when the exception propagates to the loop boundary, then it is caught, the full exception message and stack trace are logged at `Error` level, and the next iteration starts after the configured interval.
2. Given five consecutive iterations all throw exceptions, when the loop runs, then all five errors are logged and the loop continues rather than exiting.
3. Given an iteration error occurs, when the loop resumes, then the exit code of the process remains `0` (the loop completed normally), unless a fatal startup error occurred.

---

## L2-026: Working Directory Validation
**Traces to:** L1-009

Before the loop starts, the tool shall validate that the resolved working directory exists on disk and contains a `.git` directory (or is otherwise a valid git work tree as determined by `git rev-parse --is-inside-work-tree`). Validation failures shall exit the process with exit code `1`.

**Acceptance Criteria:**
1. Given `--path` resolves to a directory containing a `.git` folder, when the tool starts, then validation passes and the loop begins.
2. Given `--path` resolves to a directory without a `.git` folder, when the tool starts, then it logs `"The specified path is not a git repository"` and exits with exit code `1`.
3. Given `--path` resolves to a path that does not exist on disk, when the tool starts, then it logs `"The specified path does not exist"` and exits with exit code `1`.
4. Given the current working directory is a git repository and no `--path` is specified, when the tool starts, then validation passes and the loop begins.

---

## L2-027: Commit Message Injection Prevention
**Traces to:** L1-009

The commit message shall be passed to the `git commit` process as a direct argument (not via shell string interpolation). The tool shall use `Process.StartInfo.ArgumentList` (not `Arguments` string) to prevent shell injection via user-supplied message content.

**Acceptance Criteria:**
1. Given a commit message containing `; rm -rf /` or backtick expressions, when the git commit process is started, then the shell-sensitive characters are treated as literal text in the commit message and no shell commands are executed.
2. Given `Process.StartInfo.ArgumentList` is used, when the process is inspected, then no shell (`cmd.exe` / `sh`) is used as the host process for git invocations.
3. Given any git command is constructed, when the process is started, then `UseShellExecute` is set to `false` on all `ProcessStartInfo` instances.
